\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}
\thispagestyle{fancy}

This report presents the practical work carried out as part of the DevSecOps and Web Security lab at the National School of Applied Sciences of Tangier. The objective was to build, exploit, and secure a Flask web application while integrating automated security testing into a CI/CD pipeline.

Part 1 covered environment setup, STRIDE threat modeling, and hands-on exploitation of four critical vulnerabilities: SQL Injection, Cross-Site Scripting (XSS), OS Command Injection, and Path Traversal. Each vulnerability was analyzed, exploited against an intentionally insecure application, and subsequently remediated in a secure implementation.

Part 2 covers automated security testing using SAST (Bandit), SCA (Safety), and DAST (OWASP ZAP) tools integrated into a GitHub Actions pipeline with security gates. Part 3 addresses Docker container hardening following CIS Benchmark guidelines, runtime WAF monitoring, security headers, and an incident response exercise.

The complete lab demonstrates a full DevSecOps workflow from threat modeling to production hardening, reflecting industry best practices for secure software delivery.

\vspace{1cm}
\textbf{Keywords:} DevSecOps, Web Security, OWASP Top 10, Flask, STRIDE, SAST, DAST, Docker, CI/CD, WAF

\section{Context and Objectives}

Modern software development requires security to be integrated at every stage of the development lifecycle, not added as an afterthought. This approach, known as \textbf{DevSecOps} (Development, Security, and Operations), shifts security left by embedding automated security checks directly into the CI/CD pipeline.

This project was carried out as part of the Web Security and DevSecOps lab at the National School of Applied Sciences of Tangier. The objective was to gain hands-on experience with real-world security techniques by building, breaking, and securing a Flask web application.

The lab covers the full DevSecOps workflow: from identifying threats using structured threat modeling, to exploiting and remediating OWASP Top 10 vulnerabilities, to automating security testing in a CI/CD pipeline, and finally hardening the deployment environment against runtime attacks.

\section{Lab Structure Overview}

The lab is divided into three parts:

\begin{itemize}[leftmargin=1.5cm]
  \item \textbf{Part 1 — Build \& Break:} Environment setup, STRIDE threat modeling, and hands-on exploitation of four critical vulnerabilities in an intentionally insecure Flask application, followed by their remediation.
  \item \textbf{Part 2 — Test \& Automate:} Integration of automated security tools (Bandit, Safety, pytest, OWASP ZAP) into a GitHub Actions CI/CD pipeline with security gates that block insecure code from being merged.
  \item \textbf{Part 3 — Harden \& Monitor:} Docker container hardening following CIS Benchmark guidelines, runtime WAF and rate limiting middleware, security headers implementation, and an incident response exercise.
\end{itemize}

\section{STRIDE Methodology}

Threat modeling is a structured approach to identifying security risks before writing a single line of code. \textbf{STRIDE} is a mnemonic developed by Microsoft that categorizes threats into six categories:

\begin{table}[H]
\centering
\begin{tabularx}{\textwidth}{>{\bfseries}l l l l}
\toprule
Category & Full Name & Example Threat & Control \\
\midrule
S & Spoofing & SQL injection to bypass login & Parameterized queries \\
T & Tampering & Malicious pickle deserialization & Input validation \\
R & Repudiation & No logs — cannot trace attacker & Audit logging \\
I & Info Disclosure & API returns plaintext passwords & Access control \\
D & Denial of Service & Command injection crashes server & Rate limiting, WAF \\
E & Elevation of Privilege & Path traversal reads /etc/passwd & Least privilege, safe\_join \\
\bottomrule
\end{tabularx}
\caption{STRIDE threat categories}
\end{table}

\section{Threat Modeling Process}

The threat modeling process followed these steps:

\begin{enumerate}[leftmargin=1.5cm]
  \item \textbf{Define scope} — identify assets and trust boundaries
  \item \textbf{Create a Data Flow Diagram (DFD)} — Browser $\rightarrow$ Flask $\rightarrow$ SQLite $\rightarrow$ OS
  \item \textbf{Enumerate threats} using STRIDE for each data flow
  \item \textbf{Score each threat} using DREAD or CVSS
  \item \textbf{Prioritize and assign mitigations}
  \item \textbf{Validate mitigations} and retest
\end{enumerate}

\subsection{Exercise 1 — SQL Injection (T1)}
% ────────────────────────────────────────────────────────────────────────────

\textbf{Attack Description}

SQL Injection occurs when user-supplied input is directly concatenated into an SQL query without sanitization. An attacker can manipulate the query logic to bypass authentication, extract data, or modify the database.

\textbf{Target:} \texttt{POST /login}

\textbf{Exploitation}

The following payload was entered in the username field on the login page:

\begin{lstlisting}[style=urlstyle, caption={SQL Injection payload}]
admin' OR '1'='1
\end{lstlisting}

The injected query becomes:
\begin{lstlisting}[style=pythonstyle, caption={Resulting malicious SQL query}]
SELECT * FROM users WHERE username='admin' OR '1'='1'
\end{lstlisting}

Since \texttt{'1'='1'} is always true, the query returns the admin creedentials, completely bypassing authentication.

\subsection{Exercise 2 — Cross-Site Scripting (XSS) (T7)}
% ────────────────────────────────────────────────────────────────────────────

\textbf{Attack Description}

Reflected XSS occurs when user input is included in the HTTP response without proper escaping. An attacker can inject malicious JavaScript that executes in the victim's browser, potentially stealing session tokens or performing actions on their behalf.

\textbf{Target:} \texttt{GET /search?q=}

\textbf{Exploitation}

The following payloads were tested:

\begin{lstlisting}[style=urlstyle, caption={XSS Payload 1 — script tag}]
http://127.0.0.1:5000/search?q=<script>alert(document.cookie)</script>
\end{lstlisting}

\begin{lstlisting}[style=urlstyle, caption={XSS Payload 2 — image error event}]
http://127.0.0.1:5000/search?q=<img src=x onerror=alert(1)>
\end{lstlisting}

Both payloads triggered a JavaScript alert in the browser. The \texttt{document.cookie} payload returned an empty value because Flask sets the \texttt{HttpOnly} flag on session cookies, preventing JavaScript access. However, the XSS vulnerability itself was confirmed — in a real scenario targeting an application without \texttt{HttpOnly}, session tokens would be exfiltrated.

\subsection{Exercise 3 — OS Command Injection (T5)}
% ────────────────────────────────────────────────────────────────────────────

\textbf{Attack Description}

Command injection occurs when user input is passed unsanitized to a system shell command. Using \texttt{shell=True} in Python's \texttt{subprocess} module passes the entire string directly to the OS shell, allowing an attacker to append additional commands using shell operators.

\textbf{Target:} \texttt{GET /ping?host=}

\textbf{Windows vs WSL Context}

Initial testing was performed on Windows where Linux-specific commands were unavailable. The \texttt{ping} command on Windows uses \texttt{-n} instead of \texttt{-c}, and system files like \texttt{/etc/passwd} do not exist. The environment was migrated to WSL to replicate a realistic Linux server scenario.

On Windows, the pipe operator (\texttt{|}) was used to confirm command injection:

\begin{lstlisting}[style=urlstyle, caption={Windows command injection — whoami}]
http://127.0.0.1:5000/ping?host=127.0.0.1|whoami
\end{lstlisting}

The server returned: \texttt{desktop-v9jbra4\textbackslash electro-rimo}, confirming arbitrary command execution.

After migrating to WSL, Linux commands worked as intended:

\begin{lstlisting}[style=urlstyle, caption={WSL command injection — list home directory}]
http://127.0.0.1:5000/ping?host=127.0.0.1;ls+/home
\end{lstlisting}

\begin{lstlisting}[style=urlstyle, caption={WSL command injection — dump system users}]
http://127.0.0.1:5000/ping?host=127.0.0.1;cat+/etc/passwd
\end{lstlisting}

% ────────────────────────────────────────────────────────────────────────────
\subsection{Exercise 4 — Path Traversal (T6)}
% ────────────────────────────────────────────────────────────────────────────

\textbf{Attack Description}

Path traversal occurs when user-supplied filenames are passed directly to file system operations without validation. Using \texttt{../} sequences, an attacker can escape the intended directory and access arbitrary files on the server.

\textbf{Target:} \texttt{GET /read\_file?file=}

\textbf{Exploitation}

\begin{lstlisting}[style=urlstyle, caption={Path traversal — read system users}]
http://127.0.0.1:5000/read_file?file=../../etc/passwd
\end{lstlisting}

\begin{lstlisting}[style=urlstyle, caption={Path traversal — read application source code}]
http://127.0.0.1:5000/read_file?file=../app_vulnerable.py
\end{lstlisting}

Both requests returned the full file contents in the browser. Notably, absolute paths also worked because the vulnerable app uses Python's raw \texttt{open()} function:

\begin{lstlisting}[style=urlstyle, caption={Absolute path attack}]
http://127.0.0.1:5000/read_file?file=/etc/passwd
\end{lstlisting}

\textbf{Impact:} An attacker can read any file the application has permission to access, including configuration files, secret keys, SSL certificates, and the application source code itself. This information can be used to plan further attacks.
\end{warningbox}

\textbf{What is Secret Scanning?}

Gitleaks scans the entire git repository — including commit history — for accidentally leaked secrets like API keys, passwords, and tokens. Even deleted secrets remain in git history and can be found.

tar -xzf gitleaks_8.18.4_linux_x64.tar.gz

\begin{lstlisting}[style=bashstyle]
# Get WSL IP
hostname -I
# Use the returned IP (e.g. 172.22.159.173) in ZAP target URL
\end{lstlisting}

sed -i 's/host="127.0.0.1"/host="0.0.0.0"/' part1/app_secure.py

\section{CI/CD Pipeline with GitHub Actions}

\textbf{What is a Security Gate?}

A security gate is an automated check in the CI/CD pipeline that blocks a pull request if a security condition is not met. This enforces that insecure code can never reach production.

\subsection{Pipeline Jobs Overview}

\begin{table}[H]
\centering
\begin{tabularx}{\textwidth}{l l X l}

\textbf{Job} & \textbf{Tool} & \textbf{Purpose} & \textbf{Gate} \\
\midrule
SAST & Bandit & Scan Python code & Block if HIGH severity \\
SCA & Safety & Check dependencies & Block if critical CVE \\

Secret Scan & Gitleaks & Detect leaked credentials & Block if secret found \\
Container Scan & Trivy & Scan Docker image & Advisory only \\
DAST & OWASP ZAP & Dynamic scanning & Advisory only \\
Security Gate & Aggregator & Final merge check & Block if any job failed \\
\bottomrule
\end{tabularx}
\caption{CI/CD pipeline security gates}
\end{table}

\subsection{Problems Encountered and Fixes}

\begin{warningbox}
\textbf{Problem 1 — Invalid workflow syntax:}
The original \texttt{devsecops-pipeline.yml} used \texttt{hashFiles()} in a job-level \texttt{if:} condition, which is not supported by GitHub Actions:
\begin{lstlisting}[style=bashstyle]
# INVALID
if: hashFiles('Dockerfile') != ''

# FIX
if: true
\end{lstlisting}
\end{warningbox}

\begin{warningbox}
\textbf{Problem 2 — Wrong file paths:}
The original pipeline referenced \texttt{code/session1/} and \texttt{code/session2/} paths that do not exist in the cloned repository. All paths were updated to match the actual structure:
\begin{lstlisting}[style=bashstyle]
# ORIGINAL (wrong)
bandit -r code/session1/app_secure.py
safety check -r code/session1/requirements.txt
pytest code/session2/test_security.py

# FIXED
bandit -r app.py          # scans root app.py
safety check -r requirements.txt
PYTHONPATH=../part1 pytest part2/test_security.py
\end{lstlisting}
\end{warningbox}

\begin{warningbox}
\textbf{Problem 3 — Pipeline scanning \texttt{part1/app\_vulnerable.py}:}
Since the vulnerable app exists in \texttt{part1/}, Bandit would always find HIGH severity issues regardless of which branch was being tested.\\[4pt]

\section{Docker Security Hardening}

\textbf{What is Docker hardening?}

Docker hardening reduces the attack surface of containerized applications by applying the principle of least privilege. Even if the application code is exploited, a hardened container limits the damage an attacker can do. The provided \texttt{Dockerfile} follows CIS Docker Benchmark v1.6 recommendations.

\subsection{Dockerfile Security Controls}

The following security controls were implemented in the \texttt{Dockerfile}:

\textbf{Control} & \textbf{Implementation} & \textbf{Threat Prevented} \\
\midrule

\subsection{Problems Encountered During Docker Setup}

\begin{warningbox}
\textbf{Problem 1 — Build context missing files:}
The Dockerfile expected \texttt{requirements.txt}, \texttt{app\_secure.py}, and \texttt{templates/} in the build context, but these files were in \texttt{part1/} not \texttt{part3/}.\\[4pt]
\textbf{Fix:}
\begin{lstlisting}[style=bashstyle]
cp ../part1/app_secure.py .
cp ../requirements.txt .
cp -r ../part1/templates .
# Change build context in docker-compose.yml
sed -i 's|context: ../part1|context: .|g' docker-compose.yml
\end{lstlisting}
\end{warningbox}

\begin{warningbox}
\textbf{Problem 2 — docker-compose.yml references wrong paths:}
The original file referenced \texttt{../session1} and \texttt{../session3} which do not exist.\\[4pt]
\textbf{Fix:}
\begin{lstlisting}[style=bashstyle]
sed -i 's|../session1|../part1|g' docker-compose.yml
sed -i 's|../session3|../part3|g' docker-compose.yml
\end{lstlisting}
\end{warningbox}

\begin{warningbox}
\textbf{Problem 3 — Custom seccomp profile not supported on WSL2:}
The compose file referenced \texttt{seccomp\_profile.json} which caused a \texttt{bind-mount /proc/ns/net} error on WSL2.\\[4pt]
\textbf{Fix:} Remove the seccomp line from \texttt{docker-compose.yml}:
\begin{lstlisting}[style=bashstyle]
sed -i '/seccomp:seccomp_profile.json/d' docker-compose.yml
\end{lstlisting}
\textbf{Note:} In a production Linux environment, the custom seccomp profile would restrict available system calls, further reducing the container attack surface.
\end{warningbox}

\begin{warningbox}
\textbf{Problem 4 — gunicorn ModuleNotFoundError:}
The multi-stage build copied packages from \texttt{/root/.local} (builder stage) to \texttt{/home/appuser/.local} (production stage), but gunicorn could not find its own modules at runtime.\\[4pt]
\textbf{Root cause:} Python packages installed as \texttt{root} in builder stage are copied to \texttt{appuser} home, but the internal module references still point to the original \texttt{root} paths.\\[4pt]
\textbf{Fix:} Run \texttt{security\_monitoring.py} directly as a standalone WAF on port 5002, and use Flask's built-in server for \texttt{app\_secure.py} during testing. In production, use a single-stage build with system-wide pip install.
\end{warningbox}

\section{Runtime Security Monitoring — WAF}
% ────────────────────────────────────────────────────────────────────────────

\textbf{What is Runtime WAF?}

A Web Application Firewall (WAF) inspects incoming HTTP requests at runtime and blocks malicious payloads before they reach the application. Unlike SAST (which scans code) and DAST (which tests a running app), a WAF provides \textit{continuous} protection in production.

The \texttt{security\_monitoring.py} module implements a Flask middleware WAF with:
\begin{itemize}[leftmargin=1.5cm]
  \item Pattern-based attack detection (SQL injection, XSS, command injection)
  \item Rate limiting (blocks after 30 requests/minute per IP)
  \item Structured JSON security event logging (SIEM-compatible)
  \item Defensive HTTP security headers on every response
\end{itemize}

\subsection{Starting the WAF}

\begin{warningbox}
\textbf{Problem encountered:} \texttt{security\_monitoring.py} uses \texttt{request.remote\_addr} for IP detection, which always returns \texttt{127.0.0.1} for local requests — making it impossible to distinguish between different clients in testing.\\[4pt]
\textbf{Fix:} Added \texttt{X-Forwarded-For} header support:
\begin{lstlisting}[style=pythonstyle]
# Before
ip = request.remote_addr or "unknown"

# After
ip = request.headers.get("X-Forwarded-For",
     request.remote_addr) or "unknown"
\end{lstlisting}
This allows spoofing a specific attacker IP during incident response simulation.
\end{warningbox}

\begin{lstlisting}[style=bashstyle, caption={Starting the WAF middleware}]
cd part3
source ../.venv/bin/activate
python security_monitoring.py 2>&1 | tee security_events.log
\end{lstlisting}

\begin{warningbox}

\subsection{Step 1 — Identify the Attacker IP from Structured Logs}

The attack was simulated by sending 50 SQL injection requests with a spoofed IP header:

\begin{lstlisting}[style=bashstyle, caption={Simulating the attack}]
for i in $(seq 1 50); do
  curl -s 'http://127.0.0.1:5002/login?username=admin%27+OR+%271%27%3D%271' \
  -H "X-Forwarded-For: 203.0.113.42" > /dev/null
done
\end{lstlisting}

The attacker IP was identified by parsing the structured JSON logs:

\begin{lstlisting}[style=bashstyle, caption={Identifying attacker from WAF logs}]
grep "WAF_BLOCK" security_events.log | python3 -c "
import sys, json
for line in sys.stdin:
    if 'WAF_BLOCK' in line:
        start = line.find('{')
        if start != -1:
            event = json.loads(line[start:])
            print(f'Time: {event[\"timestamp\"]} | IP: {event[\"source_ip\"]} | Rule: {event[\"details\"][\"rule_name\"]}')
" | head -5
\end{lstlisting}

\chapter{Conclusion}
% ════════════════════════════════════════════════════════════════════════════

This lab provided a comprehensive hands-on journey through the complete DevSecOps lifecycle — from identifying and exploiting vulnerabilities, to automating security testing, to hardening a production deployment and responding to live attacks.

\section{Summary of Achievements}

\textbf{Part 1 — Build \& Break} demonstrated that the most critical web vulnerabilities are not theoretical — they are trivially exploitable in real applications. SQL injection bypassed authentication in seconds, XSS executed arbitrary JavaScript in the victim's browser, command injection gave full OS access, and path traversal exposed the application's own source code. Each vulnerability was then remediated in \texttt{app\_secure.py} following the principle of \textbf{never trusting user input}.

\textbf{Part 2 — Test \& Automate} proved that manual security testing is not enough. Bandit eliminated 100\% of HIGH and MEDIUM severity findings automatically. Safety identified 10 real CVEs in production dependencies. Gitleaks detected hardcoded secrets in git history. OWASP ZAP revealed missing security headers. GitHub Actions tied all these tools into a CI/CD pipeline that \textbf{blocks insecure code from being merged} — making security a continuous, automated practice rather than a one-time gate.

\textbf{Part 3 — Harden \& Monitor} showed that application-level security is not enough on its own. Docker hardening following CIS Benchmark guidelines ensures that even a compromised container cannot escalate privileges or persist malware. The WAF middleware provided runtime protection, detecting and blocking SQL injection, XSS, and command injection payloads in real time. The incident response exercise demonstrated a complete P1 response workflow: detect, contain, rotate credentials, document, and prevent recurrence.

\section{DevSecOps Workflow Completed}

The complete DevSecOps cycle was demonstrated end-to-end:

\begin{enumerate}[leftmargin=1.5cm]
  \item \textbf{Model} — STRIDE threat modeling identified 8 threats before writing any code
  \item \textbf{Code} — Secure implementation applied parameterized queries, safe\_join, template escaping
  \item \textbf{Scan} — Bandit, Safety, and Gitleaks caught vulnerabilities and secrets automatically
  \item \textbf{Test} — pytest security tests validated that attacks are blocked by the secure app

\section{Known Limitations and Future Work}

Several limitations were encountered during the lab that would need to be addressed in a real production environment:

\begin{itemize}[leftmargin=1.5cm]
  \item \textbf{WSL2 constraints:} Custom seccomp profiles, network namespace isolation, and some Docker security options are not fully supported on WSL2. A production deployment on native Linux would enable these controls.
  \item \textbf{gunicorn path issue:} The multi-stage Dockerfile's package path resolution failed on WSL2. A production fix would use a single-stage build with system-wide installation or a properly configured virtual environment.
  \item \textbf{WAF as middleware:} \texttt{security\_monitoring.py} runs as a standalone app rather than integrated middleware. A production WAF should be integrated at the nginx/reverse proxy level for zero-downtime enforcement.
  \item \textbf{CSRF protection:} The secure app does not implement CSRF tokens. A production Flask app should use \texttt{Flask-WTF} for CSRF protection on all form endpoints.
  \item \textbf{HTTPS:} The lab ran on HTTP. All production deployments must enforce HTTPS with valid TLS certificates and HSTS headers.
\end{itemize}
